Loading & Writing a Configuration File into OTP
About this document
Scope and purpose
The scope of this document covers the functionality of the One Time Programmable (OTP) memory space. The
purpose of the document is to provide the user with the information required to provide steps to load, write
and store configuration file’s register values into the digital controller’s OTP.
It is highly recommended that customers submit configuration file to Infineon for review and approval prior to
use in voltage regulator applications. Release of configuration files for production builds without review and
approval by Infneon may result in system failures.
Intended audience
Engineering or manufacturing users that require information on how to program the memory space of the
digital controller’s device. Please see supported controller types and versions listed in Table 1. Also listed in
the table are the different programming boards appropriate for each controller available from the Infineon
Sample Request center (ISaR).

Hardware Set-up and Connections

2

Hardware Set-up and Connections

To communicate with the controller, several I/O pins must be connected properly as shown in Figure 1. 3rd party
programmers can leave all other pins open. A default value of 49.9 k is recommended for R_Addr. A 10 Ω
resistor can be placed in series between I2C_SDA and the SDA pin of the device to improve signal integrity.
If the controller is being programmed on a system board, it is not necessary to disable the voltage regulators
during programming. The programming process will bypass the operational registers and write directly into
nonvolatile memory without interrupting operation. After programming, the new settings will take effect only
after a 3.3 V power cycle or if commanded by an NVM download commands.
Note that the I2C interface is open drain. Pull up resistors must be connected to the bus. If the device has not
been previously programmed, the pull up voltage should be 3.3 V. 1.8 V I2C logic is not support on
unprogrammed devices.

Communication with XDPE1x2xx

3.1

Basic PMBus protocols

SMBus protocols are used to communicate with XDPE1x2xx. To program the device, three particular protocols
are used. The programmer should be familiar with the WRITE_BYTE (Figure 5), BLOCK_WRITE (Figure 6), and
BLOCK_READ (Figure 7) protocols. PEC can be used for added communication reliability. A combination of
these commands is needed to successfully program the device because the data has to be communicated over
an I/O buffer.

3.2

Device address

During power up of the XDPE controller, the resistance of the R_addr resistor connected to the XAddr pin in
Figure 1 is measured. The main function of the resistor is to determine the PMBus address of the device.
Additionally, multiple configuration files can be programmed into the XDPE controller such that the controller
will automatically download the desired configuration file based on the XValent resistor value. The controller
encodes the XV resistance value into one of 16 codes ranging from 0 to 15 (0xF). The XVcode must be determine
when storing data to OTP (see chapter 5) so the controller can download the correct value during subsequent
power cycles. The encoding of the XVcode is shown in Table 2. XV0 is the default configuration XVcode. The
XDPE controller will always default to the XV0 configuration when its own XVcode configuration is not found
(not programmed) in the OTP.

3.3

MFR Specific Register Read/Write

Special manufacturer specific registers are provided to read from and write to non-standard PMBus registers in
XDPE controllers using the BLOCK_READ and BLOCK_WRITE protocols. Each non-standard PMBus register is
assigned a 32 bit register address. To access them, a user needs to first write the register address to the Register
Address Pointer (Table 3) with the BLOCK_WRITE protocol. Then the user can write to the register with the
MFR_REG_WRITE command using the BLOCK_WRITE protocol or read from the register with the
MFR_REG_READ command using the BLOCK_READ protocol.

3.4

codes to perform various tasks such as storing patches OTP or finding OTP remaining size information. Table 4

MFR_FW_COMMAND_DATA. Note that the data in the MFR_FW_COMMAND_DATA register is overwritten by
executing the MFR_FW_COMMAND.

Configuration Row number

Each succeeding row begins with a 3-digit running byte count (in hexadecimal base). It can be used as a row

First DWORD in Configuration Data header

The first DWORD in a data section also contains the XVcode and the section type information. The section type

section type is SVID partial (header code 0x11), then the most significant byte defines the associated loop (0x0,
0x1, 0x2 for loopD) and the 2nd most significant byte is the partial register command. Those two bytes should be
set to 0 for any other header codes.

5.4

Configuration Size and redundant size

The second DWORD in the configuration data header specifies the size information of the section. The size of
the section is recorded twice for data integrity purposes. In Figure 9, the size of the section is “0x0228” in
hexadecimal units which is 552 bytes of data. This is verified by the header checksum value.

5.5

Configuration Header CRC

The third DWORD in the configuration header is the header CRC. The header CRC checks the validity of the first
two DWORDs. In Figure 9, the header CRC is found to be “0x34394daa”.

5.6

Configuration Data

Following the header CRC, the register data information is all packed together in a data block. The block is
divided into DWORDs. DWORDs are used because that is the most efficient way to program the device based on
the native 32-bit architecture of the communication interface to internal registers. As shown in Figure 10, the
DWORD ordering in the configuration file is DWORD0 on the left, followed by DWORD1, DWORD2, and finally
DWORD3 is at the furthermost right location on each row. Note that the byte ordering within a DWORD is such
that the most significant byte (bits [31:24]) is located on the left and the least significant byte (bits [7:0]) is on
the right. When programming the device, the entire DWORD would be programmed at the same time.

5.8

Special considerations for partial configuration data

Some special considerations must be made to deal with partial configuration data. Partial configuration data
can be used to override some of the registers in a basic configuration file based on the R_addr value connected
to the XADDR pin (see Figure 1). Different values of this resistor will decode to 1 of 16 different XVcodes inside
the controller (see Table 2 - XVcode). This feature allows the programming multiple rails needing different
configuration settings in a given project with the same exact configuration file, thereby simplifying the
programming procedure or reducing the device part numbers if ordering preprogrammed parts from factory.
For example, on a dual CPU board, each CPU may require slightly different configuration settings due to layout
differences. If the devices are to be programmed prior to installation on the board, each programmed part
needs to be tracked to make sure it is placed to the designated CPU rail. By using partial configuration, the
differences between the two configurations are programmed in all devices so that any device can be placed on
either rail. The resistor value from the board connecting to the XADDR will determine the final configuration
settings.
Unlike a full configuration where the size of the section is predefined, the differences in settings between the
two rails depend on the actual application so that the size of the partial configuration sections can vary from
project to project. Utitilizing partial configuration instead of using full configurations is the savings of OTP
space.

5.8.1

Partial varieties

As can be see from Table 7, XDPE1x2xx controllers recognize three different types of partial configuration data,
namely Config Partial, PMBus Partial, and SVID/SVI3 Partial modifying the Config, PMBus LoopA/B, and
SVID/SVI3 LoopA/B/D sections respectively.

5.8.1.1

Config Partial

The format of the Config Partial is very similar to the other aforementioned sections: a commented section
header with an XVcode designation followed by data header, data and data CRC. The size and redundant size
information reside in same location: second DWORD in the data header on Row 000. Note that the size of this
config partial section must be used by users to program and upload to OTP (see chapter 6.4).

5.8.1.2

PMBus Partial Section

PMBus partial section is divided into individual PMBus commands. Each PMBus command has its own
configuration header, size, header CRC, data and data CRC. Up to 32 PMBus commands are allowed per XVcode.
The number of bytes per command is the number of data bytes for the PMBus command itself which ranges
from 1 byte to 31 bytes plus the bytes occupied by the headers and CRCs. This sum is rounded up to the nearest
4-byte boundary resulting in the partial data size range to be from 20 bytes to 48 bytes per command.
Invalidating a PMBus partial section invalidates all PMBus commands in that section. All PMBus commands in
the same section should be uploaded at the same time

As the case of PMBus partials, each register will restart with row 000. Since 20 bytes do not fill up the 2 rows

register, the total upload size in the section will be number of partial registers x 20 bytes.
As the case with PMBus partials, only 32 SVID/SVI3 partial commands can be saved for a given XVcode.

6.5

Error checking

It is recommended to check for upload errors after each upload wait period. If the upload is not successful, then
steps 6.2…6.4 should be repeated. If the upload is not successful, the XDPE controller will generate a CML Other
Memory Fault status. Bit[0] of the PMBus STATUS_CML (0x7e) register will be asserted indicating this fault. It is
recommended that if the device fails to be programmed after two consecutive attempts, the device should be
returned to the factory for failure analysis.
// reading status faults after programming
// output bytes: d0 where d0 is the returned cml status
0010 READ_BYTE(PMB_Addr, 0x7e, d0)
If bit 0 of d0 is not 0, it would indicate an unsuccessful upload. The user should retry to program the section
again by repeating the steps in section 6.2…6.4.
For a more thorough check of the success of the upload, a download of the OTP data is needed so that registers
can be read back individually to compare against the configuration file. A reset of the controller is needed to
download the configuration. See chapter 10.3.

7

Data section specific upload commands

Chapter 6 describes the generic procedure to upload any configuration section through the scratchpad.
Programming through the scratchpad has the main advantage of duplicating data exactly onto the OTP.
Therefore, it is the recommended method to program a configuration file to a device. For each section type of
data, XDPE controllers do offer alternative programming PMBus commands or manufacture specific PMBus
commands to save data to OTP. With data section specific upload commands, there is no need to invalidate old
data as the controller will invalidate it automatically.

7.1

Base Configuration Uploads

Base configuration sections include the Config section, the PMBus LoopA/B sections, and the SVID/SVI3
LoopA/B/D sections.

7.1.1

Section type: Config

If internal Config registers have already been programmed, users can use the MFR_FW_COMMAND
OTP_CONFIG_STORE to save the data to OTP. Suppose a device has already been programmed and mounted, it
will download the configuration during device power up. If a user makes modifications to the data in the Config
section, change the LBuck setting for example, the user does not need to reprogram the whole section
manually to save the data. The user only needs to send the following commands to upload the register data to
OTP. Note that although the invalidation step is not necessary, the user still needs to specify the XVcode.

Checksum

After all uploads have completed, the system CRC can be verified. The MFR_FW_COMMAND_GET_CRC
command is provided to help validate proper storage of configuration data to OTP. Based on the type of
validation requested, the MFR_FW_COMMAND_GET_CRC uses different algorithms to verify the data.

8.1

Total configuration checksum

In the configuration file under the Basic device information, the Configuration Checksum is listed for
verification. The Configuration Checksum is calculated by searching for all available sections in the OTP
(excluding trims). For each section, both the header CRC and the data CRC are recorded. Regardless of file
types such as USER PMBus or Partial Data or XVcode designations, all CRC are then sum together keeping only
the lower 32 bits on overflow. The advantage of this algorithm is that the order of the CRC does not matter in
the file and each CRC is included in the calculation.
The Configuration Checksum from the device can be retrieved using the MFR_FW_COMMAND GET_CRC:
// retrieve the configuration checksum from the device:

wait 20 ms

8.3

Partial sections include Config partial (0xA), PMBus partial (0xB), SVID or SVI3 partial (0x11). As multiple partials
can exist within a configuration file, there is a possiblility that one partial section can write over the data from
another partial section. Hence the order of the partial data appearing in the OTP can affect the programming of
the device. The simple summation of the header and data CRCs cannot resolve this issue. Hence the
MFR_FW_COMMAND GET_CRC command uses the standard CRC-32 algorithm computing upon the header and
data CRCs themselves. For each matching header and XVcode section as it appears in the order saved in the
OTP, the 4 bytes of the header CRC is fed into the CRC calculator (little endian) followed by the 4 bytes of the
data CRC (see chapter 8.3.1 for the CRC parameters). After all matching header code and XVcode have been
entered, the resultant CRC is returned.

10.1

Reading OTP data

To debug programming issues, it is very useful to read back the OTP data to identify the problems. Using the
MFR_REG_READ with the BLOCK_READ command, OTP data can be readily retrieved. The address for the
partition 0 of the OTP is 0x10020000. By pointing the Register Pointer RPTR to this address, the entire OTP can
be navigated.
As mentioned in chapter 5, all configuration sections follow the same format. The header code DWORD
identifies the type of data and the size DWORD indicates the size of the section. By pointing the RPTR to the top
of the OTP 0x10020000 and reading the first two DWORDs, the first section and its size can be retrieved. If this is
the section of interest, the entire section can be read out by using the MFR_REG_READ command. As the case
for MFR_REG_WRITE command, each time the MFR_REG_READ command is executed the target pointer will
automatically increment to the next address. Therefore, by continuously executing the MFR_REG_READ
command (size – 8)/4 more times, the entire first section is read. Note that since the first 2 DWORDs have
already been read out (header code DWORD and size DWORD), the remaining data size is size – 8.
If the first section is not of interest after reading the first two DWORDs, the target pointer can be reassigned to
the next section by setting the RPTR to 0x10020000 + size. Two more DWORDs are read and examined. This
process can be repeated until all OTP sections are read.
